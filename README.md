# Claude apps gateway on AWS (Bedrock upstream)

A self-hosted deployment of the **Claude apps gateway** — the enterprise
auth/telemetry gateway built into the `claude` CLI (`claude gateway`) since
2.1.195 — running on AWS ECS Fargate with a Bedrock upstream and SSO sign-in.

## What the gateway is

Developers point Claude Code at the gateway's base URL (via managed settings).
The gateway:

- Authenticates the developer with **OIDC SSO** (RFC 8628 device flow) — no
  per-developer cloud credentials.
- Holds the single **upstream credential** (here, the ECS task role's Bedrock
  access) and forwards inference (`POST /v1/messages`) to **Amazon Bedrock**,
  translating model ids (`claude-opus-4-8` → `us.anthropic.claude-opus-4-8`).
- Enforces a server-side **model allowlist**, per-domain access
  (`allowed_email_domains`), and emits an **audit log** (`session.mint`,
  `inference`, …).

The gateway is **fail-closed at boot**: bad config, unreachable Postgres, failed
OIDC discovery, or a bad upstream all exit the process. Inference creds resolve
on the first request, not at boot.

## Architecture (what was deployed)

```
                          AWS account <AWS_ACCOUNT_ID> / us-west-2
                          Default VPC <VPC_ID> (172.31.0.0/16)

   developer (in-VPC)
        │ HTTPS (CA-trusted, host gw.claude-gateway.internal)
        ▼
   ┌─────────────────────────┐   Route53 private zone claude-gateway.internal
   │ internal ALB (IPv4 only) │   → A record gw.* → ALB private IPs (RFC1918)
   │  443 HTTPS  ACM leaf     │   ip_address_type=ipv4 (NO public AAAA → /login OK)
   └─────────┬───────────────┘
        /dex*│            │/* (everything else)
             ▼            ▼
      ┌────────────┐  ┌───────────────────────────┐
      │ Dex (ECS)  │  │ gateway (ECS Fargate ARM64)│
      │ OIDC IdP   │  │ claude 2.1.196 native      │──► Amazon Bedrock (us-west-2)
      │ :5556      │  │ :8080  task role: bedrock  │    InvokeModel[WithResponseStream]
      └────────────┘  └─────────────┬─────────────┘
                                    │ sslmode=require
                                    ▼
                          ┌──────────────────────┐
                          │ RDS PostgreSQL 16.14  │ (private, not public)
                          │ db.t4g.micro          │ gateway store + migrations
                          └──────────────────────┘

   Secrets Manager: oidc-client-secret, jwt-secret, postgres-url
   CloudWatch logs: /ecs/claude-gateway (7-day retention)
```

- **Compute:** ECS Fargate, ARM64, cluster `claude-gateway`. Two services:
  `gateway` (1 task) and `dex` (1 task).
- **Images (ECR):** `claude-gateway:2.1.196` (base, native binary),
  `claude-gateway:2.1.196-gw` (base + baked `gateway.yaml` + CA entrypoint),
  `claude-gateway-dex:v2.41.1` (Dex + baked config + secret-substitution
  entrypoint).
- **IdP:** self-hosted **Dex** v2.41.1, static user `dev@example.com`, static
  client `claude-gateway`. Issuer `https://gw.claude-gateway.internal/dex`.
- **TLS:** one self-signed CA + leaf (SAN `gw.claude-gateway.internal`),
  generated by the Terraform `tls` provider, leaf imported to ACM and attached
  to the ALB HTTPS listener. CA in `certs/ca.pem`.
- **IaC:** Terraform under `terraform/` (local state, gitignored). Account-specific
  values (`vpc_id`, `subnet_ids`) are placeholders in `variables.tf` — set the
  real values in a gitignored `terraform/terraform.tfvars`.

## Developer connect steps

1. Trust the gateway CA: `export NODE_EXTRA_CA_CERTS=/path/to/certs/ca.pem`.
2. Install managed settings (`verify/managed-settings.json`):
   ```json
   { "forceLoginMethod": "gateway",
     "forceLoginGatewayUrl": "https://gw.claude-gateway.internal" }
   ```
   Path: `/Library/Application Support/ClaudeCode/managed-settings.json` (macOS)
   or `/etc/claude-code/managed-settings.json` (Linux).
3. On first connect the CLI fingerprints the TLS leaf and prompts to trust it.
   Expected SHA-256:
   `28:D1:EE:2D:C2:D5:4A:D3:6F:F5:36:A5:8E:21:C4:E9:CF:BA:0C:0B:08:98:B3:E3:2D:32:40:06:6B:34:25:8A`
4. `claude /login` → the device flow opens the verification URL → sign in at Dex
   with `dev@example.com` / `gateway-dev-pw` → approve → the CLI gets a bearer
   token and can run inference.

> The internal ALB resolves to **private IPs only**, so all of this runs from
> inside the VPC (e.g. the provisioning EC2 host). The gateway's `/login` guard
> requires the host to resolve to private addresses.

## How to verify

```bash
# 1. End-to-end SSO device sign-in -> mints a gateway token (/tmp/gw_token.txt)
bash verify/e2e-signin.sh

# 2. Inference through the gateway from Bedrock
bash verify/inference-check.sh        # prints: gateway-bedrock-ok

# 3. Auth surface + readiness
curl -sS --cacert certs/ca.pem https://gw.claude-gateway.internal/readyz
curl -sS --cacert certs/ca.pem https://gw.claude-gateway.internal/.well-known/oauth-authorization-server | jq

# 4. Audit log
aws logs filter-log-events --region us-west-2 --log-group-name /ecs/claude-gateway --filter-pattern 'session.mint'
aws logs filter-log-events --region us-west-2 --log-group-name /ecs/claude-gateway --filter-pattern 'inference'
```

## How to tear down

```bash
bash teardown.sh --dry-run   # terraform plan -destroy, no changes
bash teardown.sh --yes       # terraform destroy + delete ECR repos
```

## Resilience notes

- ALB health check targets **`/healthz`** (liveness), not `/readyz`. Rationale
  from the deploy guide: `/readyz` removes replicas during a Postgres outage
  (fail-closed for new + existing); `/healthz` keeps already-signed-in
  developers working through a transient store blip while only new sign-ins
  fail. For a single-task test deployment `/healthz` is the safer default.
- `health_check_grace_period_seconds=120` lets the gateway run DB migrations on
  first boot before the ALB starts failing it.
- CloudWatch log retention: 7 days. Desired task count: 1 (raise for HA).

## Security posture

- Task role grants only `bedrock:InvokeModel` +
  `bedrock:InvokeModelWithResponseStream` on scoped `anthropic.*` /
  `us.anthropic.*` ARNs — no wildcards, no `Resource: *`.
- RDS is `PubliclyAccessible=false`, `sslmode=require`, encrypted.
- Security groups: ALB ← VPC CIDR (443); tasks ← ALB SG (8080/5556); RDS ← task
  SG (5432).
- All secrets live in Secrets Manager and are injected as env; configs use only
  `${ENV}` placeholders. `certs/`, `*.tfstate`, `*.tfvars`, the native binary
  are gitignored.
- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1` set in the gateway container.
